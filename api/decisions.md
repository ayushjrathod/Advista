Option 1 — Agent with get_brief/update_brief tools
  - With Groq's smaller models, tool-calling reliability is lower — the agent may skip calling update_brief even when it should
  - You get unpredictable turn costs: some turns trigger 2-3 tool round-trips, others zero
  - The agent can hallucinate field values into update_brief with false confidence (especially for competitors the user never named)
  - Adds orchestration complexity for what is a fundamentally linear form-filling task

  Option 2 — Full ResearchBrief + response as structured output
  - Forces the LLM to be conversational AND fill a complete 8-field struct simultaneously — these objectives fight each other, degrading both
  - The LLM will guess/hallucinate empty fields to satisfy the schema (Pydantic requires values)
  - You lose the natural "I don't know this yet" state — every response contains a full brief, most of it empty or fabricated

  ---
  Better approach: Delta structured output
  
  Return only what was learned this turn, not the full brief:

  class TurnOutput(BaseModel):
      response: str               # conversational reply to user
      brief_updates: dict         # only fields learned in this exchange

  Single LLM call per turn. You merge brief_updates into your stored brief. The LLM only fills keys it actually learned — everything else is absent (not
  empty-string guessed). Context window stays fixed (conversation history only, not a growing re-extraction prompt).

----

LLM helper — class vs functions (api/src/utils/llm.py)

  Use a class because both generate() and generate_structured() share the same
  initialized state: client, model, project, region. The client is expensive to
  initialize and should be created once. Passing those four values as parameters
  to every function call would be worse.

  Rule of thumb applied here:
    - Class  → multiple methods share initialized state / lifecycle
    - Function → fully self-contained, all inputs come in as parameters

----

BriefUpdates schema — derived from ResearchBrief, not duplicated (api/src/services/chat_service.py)

  BriefUpdates is created dynamically via Pydantic's create_model, making all ResearchBrief
  fields Optional[T] = None. This means:
  - No duplication — add a field to ResearchBrief and BriefUpdates picks it up automatically
  - model_dump(exclude_none=True) gives only the fields the LLM actually learned this turn
  - Using plain dict instead would make the JSON schema too loose for Gemini structured output,
    causing the LLM to return {} every time

----

Session-based memory — singleton service, dict of sessions (api/src/services/chat_service.py)

  ChatService is a module-level singleton. It holds sessions: dict[str, Session] where each
  Session is a Pydantic model containing memory (conversation history) and brief.

  Session ID is generated by the backend on the first message (not by the frontend). The ID is
  returned in every response so the frontend can reuse it. Omitting session_id creates a new session.

  Limitation: sessions live in-memory on one process. With multiple uvicorn workers, a request
  hitting a different worker won't find the session. Move to Redis when scaling.

----

Competitor search enrichment — SerpAPI + structured extraction, not Gemini grounding

  Gemini's structured output (response_schema) and grounding tools (google_search, url_context,
  function calling) are mutually exclusive — the API returns an error if both are set.

  Workaround: search_query field in TurnOutput acts as a manual tool call signal.
  When the LLM sets it, the code:
    1. Calls SerpAPI directly (no LLM) → raw title+snippet text
    2. Calls generate_structured(response_schema=BriefUpdates) on that text → enriched fields
    3. Merges into brief silently — user sees only the conversational response

  Considered using Gemini grounding for search, but that adds an extra LLM call (grounding
  synthesizes results via LLM before returning text). SerpAPI skips that synthesis step.

----

URL pre-fill — httpx fetch + structured extraction, not URL context grounding tool

  Same structured output conflict applies: url_context tool cannot be combined with response_schema.
  Solution: fetch the URL with httpx directly, pass the first 8000 chars of HTML to
  generate_structured. One LLM call, no grounding dependency, simpler and more predictable.
